## 任务 1：API Server graceful shutdown（P0，30 分钟）

**文件**：`artifacts/api-server/src/index.ts`

**问题**：现在收到 SIGTERM 立即 `py.kill()` + `process.exit(0)`，所有正在跑的长流式请求被硬切，用户看到"开了流不出字"。

**改动**：把现有

```ts
process.on("exit", () => py.kill());
process.on("SIGTERM", () => { py.kill(); process.exit(0); });
process.on("SIGINT",  () => { py.kill(); process.exit(0); });
```

替换为：

```ts
let shuttingDown = false;
let httpServer: any = null;  // app.listen 的返回值，需要把 app.listen 改成保留 server 引用

async function gracefulShutdown(signal: string) {
  if (shuttingDown) return;
  shuttingDown = true;
  logger.info({ signal }, "Graceful shutdown initiated");

  // 1) 停止接受新连接（保留 in-flight）
  if (httpServer) httpServer.close(() => logger.info("HTTP server closed"));

  // 2) 给 Python 发 SIGTERM 等它处理完当前请求；最多 30s 强 KILL
  try { py.kill("SIGTERM"); } catch {}
  await new Promise<void>((resolve) => {
    const timer = setTimeout(() => { try { py.kill("SIGKILL"); } catch {} resolve(); }, 30_000);
    py.once("exit", () => { clearTimeout(timer); resolve(); });
  });

  process.exit(0);
}

process.on("SIGTERM", () => gracefulShutdown("SIGTERM"));
process.on("SIGINT",  () => gracefulShutdown("SIGINT"));
```

把文件底部的 `app.listen(port, ...)` 改成：

```ts
httpServer = app.listen(port, (err) => { ... });
```

**验证**：本地 `kill -TERM <pid>` Express 进程，应看到日志依次出现 `Graceful shutdown initiated` → `HTTP server closed` → 进程退出。开个长流式 chat 同时 SIGTERM，流应该正常跑完而不是被斩断。

## 任务 3：`_get_low_personal_key` 5 行绷带修复（P0，最小改动）

**文件**：`jetbrainsai2api/main.py`，找到 `def _get_low_personal_key(discord_id: str) -> str:`（约 line 9604）

**问题**：会把 LOW 用户名下的孤儿预签 key 误当作个人 key 返回，导致批量激活的额度发到错误对象上。

**改动**：替换整个函数：

```python
def _get_low_personal_key(discord_id: str) -> str:
    """从内存中查找 LOW 用户的个人专属密钥（按 low_admin_discord_id 匹配）。
    返回 key 字符串，未找到返回空字符串。
    
    优先级（避免命中预签 key 等孤儿）：
      1. usage_count > 0（已被使用过的 key 更可能是真个人 key）
      2. usage_limit > 0（有真实额度）
      3. 没有上述特征则按字典顺序兜底
    """
    if not discord_id:
        return ""
    candidates = [
        (k, v) for k, v in VALID_CLIENT_KEYS.items()
        if v.get("low_admin_discord_id") == discord_id
    ]
    if not candidates:
        return ""
    candidates.sort(key=lambda kv: (
        -int(kv[1].get("usage_count") or 0),
        -int(kv[1].get("usage_limit") or 0),
    ))
    return candidates[0][0]
```

**验证**：
- `pytest` 单元测试（如有）跑一遍
- 模拟一个 LOW 用户：先调 `/admin/activate`（产生孤儿预签 key），再调 `/admin/low-user-key` POST 创建真 personal key（usage_limit=0），再调 `/admin/activate-batch` → 累加额度应该上到真 personal key 而不是预签 key。
- 不需要 schema 变更或迁移。

2. 加后台任务：

```python
async def _cf_proxy_health_check_loop():
    """每 5 分钟探活所有 CF Worker URL，连续 3 次失败自动 is_active=false"""
    while True:
        try:
            await asyncio.sleep(300)
            pool = await _get_db_pool()
            if not pool:
                continue
            async with pool.acquire() as conn:
                rows = await conn.fetch(
                    "SELECT id, url, consecutive_failures FROM cf_proxy_pool WHERE is_active = TRUE"
                )
            
            async with httpx.AsyncClient(timeout=8) as client:
                for row in rows:
                    ok = False
                    try:
                        r = await client.get(f"{row['url']}/health")
                        ok = (r.status_code == 200)
                    except Exception:
                        ok = False
                    
                    async with pool.acquire() as conn:
                        if ok:
                            await conn.execute(
                                "UPDATE cf_proxy_pool SET last_health_check=$1, consecutive_failures=0 WHERE id=$2",
                                time.time(), row["id"]
                            )
                        else:
                            new_fails = (row["consecutive_failures"] or 0) + 1
                            if new_fails >= 3:
                                await conn.execute(
                                    "UPDATE cf_proxy_pool SET is_active=FALSE, last_health_check=$1, consecutive_failures=$2 WHERE id=$3",
                                    time.time(), new_fails, row["id"]
                                )
                                print(f"[cf-health] 自动停用 {row['url']}（连续失败 {new_fails} 次）")
                            else:
                                await conn.execute(
                                    "UPDATE cf_proxy_pool SET last_health_check=$1, consecutive_failures=$2 WHERE id=$3",
                                    time.time(), new_fails, row["id"]
                                )
            
            # 触发重新加载内存池
            await load_cf_proxies_from_db()
        except Exception as e:
            print(f"[cf-health] loop error: {e}")
```


## 任务 7：CF Worker 支持手动重定向（让 OAuth 也走代理，P2）

**文件**：`cf-worker.js`

**问题**：当前 Worker 写死 `redirect: "follow"`，吃掉所有 302 → OAuth PKCE 必须直连 → 登录 IP 没分散。

**改动**：替换 fetch 调用部分：

```javascript
// 在原来 const response = await fetch(targetUrl, {...}); 之前
const redirectMode = request.headers.get("x-redirect-mode") || "follow";
const validRedirectModes = ["follow", "manual", "error"];
const finalRedirectMode = validRedirectModes.includes(redirectMode) ? redirectMode : "follow";

const response = await fetch(targetUrl, {
    method: request.method,
    headers,
    body: request.method !== "GET" && request.method !== "HEAD" ? request.body : undefined,
    redirect: finalRedirectMode,  // ★ 改这里
});

// 手动重定向时透传 Location
if (finalRedirectMode === "manual" && response.status >= 300 && response.status < 400) {
    const responseHeaders = new Headers();
    for (const [k, v] of response.headers.entries()) {
        const kl = k.toLowerCase();
        if (kl === "set-cookie") {
            responseHeaders.append("set-cookie", stripCookieDomain(v));
        } else {
            responseHeaders.set(k, v);
        }
    }
    return new Response(null, { status: response.status, headers: responseHeaders });
}
// 后续逻辑保持原样
```

## 任务 8：前端重试 + jitter（P3，10 行）

**文件**：`artifacts/admin-panel/src/App.tsx`

**问题**：`retry: failureCount < 5` 在 503 时多个 tab 同时打开会放大流量。

**改动**：替换 `queryClient` 的配置：

```ts
const queryClient = new QueryClient({
  defaultOptions: {
    queries: {
      retry: (failureCount, error: any) => {
        const status = error?.status ?? error?.response?.status;
        if (status === 503 || status === 502) return failureCount < 3;  // 5 → 3
        return failureCount < 1;
      },
      retryDelay: (attempt) => 
        Math.min(1000 * 2 ** attempt + Math.random() * 1000, 15000),  // 加 jitter
      staleTime: 30_000,
      gcTime: 5 * 60_000,
      refetchOnWindowFocus: false,
      refetchOnReconnect: true,
    },
  },
});
```

**验证**：本地开 5 个 tab 同时打开管理面板，故意让 Python 5 秒内返回 503，应不再看到上游被重试洪水击穿。
